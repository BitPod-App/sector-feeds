# LINEAR_BEST_PRACTICES_GUIDE_V1

Status: ACTIVE
Owner: CJ

Purpose: Operational best practices that Taylor must follow when managing Linear.
Style: Guidelines with hard minimums for M-9 bootstrap mode.
Primary external source: https://linear.app/method/introduction

## Source alignment
This guide maps directly to Linear Method principles and practices, including:
- create momentum (cycles)
- aim for clarity
- keep manageable backlog
- specify owners
- scope issues small
- measure progress with actual work
- write a changelog

## 1) Keep Linear small and legible
- Prefer fewer labels over taxonomy completionism.
- Add a label only if it solves repeated friction.
- If a label is rarely used, it is likely noise.

## 2) Every issue should answer 3 questions
1. What problem are we solving?
2. What does done look like (acceptance criteria)?
3. Who owns it right now?

If any are missing, the issue is not ready.

## 3) Bias toward short acceptance checklists
- 3-7 bullets is the sweet spot.
- If it needs 30 bullets, split it into smaller issues.

## 4) Use statuses as truth
- Status reflects reality.
- Do not move issues to In Progress until someone is actively working.

## 5) Do not use Linear as a notebook
- Keep issue bodies action-oriented.
- Link docs for long-form planning.

## 6) Make blocking explicit
- If blocked, label it.
- Every blocked issue should include a next action.

## 7) Hygiene beats heroics
Weekly (15-30 min):
- close stale tickets
- merge duplicates
- add missing acceptance criteria
- resolve blocked items into decisions

## 8) Avoid premature specialization
M-9 focus:
- Taylor + Vera + Engineer A/B workflow stability first.
- Add more specialists after a stable end-to-end loop.

## 9) Change control without slowing down
- Additive-first changes are preferred.
- Destructive schema changes require change proposal + rollback note.

## 10) Taylor principle
Optimize for:
- clarity
- sequencing
- falsifiable progress
- minimal drift

Do not optimize for:
- perfect process
- perfect taxonomy
