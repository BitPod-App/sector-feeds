# LINEAR_BEST_PRACTICES_GUIDE_V1

Status: DRAFT  
Owner: CJ  
Last updated: 2026-02-28  
Purpose: Non-strict best practices Taylor should internalize and apply.  
Style: Guidelines, not commandments.

---

## 1) Keep Linear small and legible
- Prefer fewer labels over taxonomy completionism.
- Add a label only if it solves repeated friction.
- If a label is rarely used, it’s probably noise.

---

## 2) Every issue should answer 3 questions
1) What problem are we solving?
2) What does “done” look like (acceptance criteria)?
3) Who owns it right now?

If any are missing, the issue is not ready.

---

## 3) Bias toward short acceptance checklists
- 3–7 bullets is the sweet spot.
- If it needs 30 bullets, it’s an epic masquerading as a ticket.

---

## 4) Use statuses as truth, not aspiration
- Status reflects reality.
- Don’t move tickets to “In Progress” until someone is actually working.

---

## 5) Don’t treat Linear as a notebook
- Don’t dump unstructured brainstorming into issue bodies.
- Use linked docs for big thinking; keep Linear for action.

---

## 6) Make blocking explicit
- If something is blocked, label it.
- Every blocked ticket should have a next action (decision needed / dependency / question).

---

## 7) Hygiene beats heroics
Weekly (15–30 minutes):
- close dead tickets
- merge duplicates
- add missing acceptance criteria
- resolve blocked items into decisions

---

## 8) Avoid premature specialization
Week 1 focus:
- Taylor + QA + 1–2 engineer agents
Future:
- UI/UX, FE, Brand specialists only after the pipeline is proven end-to-end.

---

## 9) Change control without slowing down
- Additive-first changes are almost always safe.
- Destructive changes: require a short Change Proposal and a rollback note.

---

## 10) “Taylor principle”
Optimize for:
- clarity
- sequencing
- falsifiable progress
- minimal drift

Not for:
- perfect process
- perfect taxonomy
